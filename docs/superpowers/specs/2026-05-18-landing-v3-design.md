# SmoothScroll Landing v3 — App-consistent, Conversion-first Redesign

> **Created:** 2026-05-18
> **Status:** Ready for implementation planning
> **Successor to:** `docs/landing/` (v2, currently live on gh-pages)

## Goal

Replace the current static landing v2 with a Next.js + shadcn/ui implementation that
visually matches the SmoothScroll desktop app and is engineered around a single
conversion goal: **a first-time visitor downloads the app within 60 seconds.**

Two non-negotiable user-stated priorities:

1. Visual consistency with the desktop app (so users do not perceive marketing site
   and product as two different things).
2. UX quality — smooth animations, considered interactions, no friction.

## Non-goals (YAGNI)

The following are deliberately excluded:

- Theme showcase carousel (does not drive download)
- Competitor comparison table (factual maintenance burden, legal exposure)
- User testimonials (no real quotes available; fabricated quotes harm trust)
- Blog or changelog page
- Vercel Analytics / Google Analytics (zero-telemetry is a selling point)
- A/B test framework
- Cinematic video hero (perf penalty without justification)

## Architecture

### Tech stack

- **Framework:** Next.js 14, App Router, `output: 'export'`
- **UI primitives:** shadcn/ui (Radix + CVA + tailwind-merge), same versions as the app
- **Animations:** Framer Motion 11 (`useReducedMotion` honored throughout)
- **Icons:** lucide-react (consistent with app)
- **Build target:** Static HTML/CSS/JS deployable to GitHub Pages
- **i18n:** Plain JSON dictionaries per locale (`en.json`, `vi.json`, `zh.json`),
  consumed via a small custom hook — no runtime i18n library, keeps bundle small

## Visual design tokens

### Color: app-consistent neutral + one marketing accent

Copy verbatim from `src/index.css` of the app (HSL hue 240 zinc/neutral):

```css
:root {
  --background: 0 0% 100%;
  --foreground: 240 10% 3.9%;
  --card: 0 0% 100%;
  --primary: 240 5.9% 10%;
  --muted: 240 4.8% 95.9%;
  --border: 240 5.9% 90%;
  --radius: 0.5rem;
}
.dark { /* identical mapping to app */ }
```

Plus a single marketing accent layer used only for hero CTA and the highest-priority
headline accents:

```css
:root {
  --brand-from: 220 90% 65%;
  --brand-to:   260 80% 70%;
  --brand-grad: linear-gradient(135deg, hsl(var(--brand-from)), hsl(var(--brand-to)));
}
```

The accent is restricted to:

- The primary download button background
- One hero word styled with `bg-clip-text`
- The sticky bottom bar's CTA
- Active install-tab indicator

Everywhere else is monochrome neutral, matching the app.

### Typography

- Font stack: system stack (matches app)
- Base size: **16px** on the landing (vs 13px in the app)
- Headings: tight letter-spacing `-0.02em`, font-weight 700
- Body: 1.6 line-height, max-width 65ch

### Spacing & radius

Same scale as app: `--radius: 0.5rem`. Section vertical rhythm: 80–96px on desktop,
48–64px on mobile.

### Motion principles

- **Default duration:** 200ms with `cubic-bezier(0.16, 1, 0.3, 1)`
- **Spring physics** for interactive elements. Damping 25, stiffness 300.
- **`useReducedMotion`** disables all transforms; opacity-only fallback retained
- **No parallax**, no infinite-scroll loops on hero

### Hero + DemoScroll

The hero is the conversion linchpin. Two columns on desktop, stacked on mobile:

**Left column:** eyebrow → H1 with one accent word → subhead → primary CTA
(auto-detect OS) → trust line → secondary ghost button "See how it works".

**Right column:** `DemoScroll` component. A 480-pixel-tall scrollable card
containing ~30 lines of representative content. A `Switch` above the card toggles
between native scroll behavior and a JS-driven smoothed scroll using the same
easing curve as the app's default. Prompt: "Scroll the card. Then flip the switch."

Heuristic instrumentation (no analytics, just local logic):

- If the user toggles the switch and then scrolls the card again, the hero CTA
  receives a one-off pulse animation and a small toast appears: "Want to feel it
  for real? — Download." Once per session only.

### TrayPreview component

A pixel-faithful interactive recreation of the app's tray panel, not a screenshot.
Built from real shadcn components so it shares styling with the actual product:

- Header with app icon + version
- "Smoothing strength" slider that reacts and shows the current value
- Theme switcher (3 of the app's 7 themes — kept small for bundle size)
- Hotkey hint
- "Settings…" / "Quit" footer row

This achieves the "looks like the same product" goal more strongly than any static
screenshot, because the user touches the actual UI.

### Stats

Build-time fetched, client-side hydrated:

- GitHub stars
- Total downloads across releases (sum of `asset.download_count`, capped at last 5
  releases)
- Latest version

If any fetch fails: hide that stat rather than show "—" or "N/A".

### Install section

Two tabs (Windows / macOS). Each tab content:

- Numbered three-step flow with screenshots
- Code-block-style box containing the actual installer filename + copy button
- One install-note (`%LOCALAPPDATA%` for Windows, Accessibility permission for Mac)

Tab default selection respects user OS detection.

## Conversion mechanics

A single hook owns download URL logic; all CTAs consume it:

```ts
function useDownloadUrl(): {
  url: string;
  version: string;
  os: 'win' | 'mac' | 'other';
  sizeLabel: string;
  ctaLabel: string;
}
```

- On first render, returns fallback URL using bundled-at-build-time release data
- After mount, refetches `api.github.com/repos/.../releases/latest` and updates
  state — page interactive immediately, accuracy improves once network returns
- All CTAs read from this single hook, so wiring stays DRY

Multi-touch CTA orchestration:

| Trigger | Effect |
|---|---|
| Page mount | Hero CTA visible, OS-detected |
| User toggles `DemoScroll` then scrolls card | Hero CTA pulses once + toast (max once per session) |
| ScrollY > hero bottom | Sticky bottom bar slides up |
| Mouse leaves viewport top + first time this session | Exit-intent modal |
| User clicks any CTA | Open download URL in new tab + dismiss sticky bar |
| User dismisses exit-intent modal | Suppress for this session (localStorage) |

## Data flow

### Release data (build time)

```
next build  →  generateStaticParams() per locale  →
  fetch('api.github.com/repos/.../releases/latest')  →
  inline tag_name, assets[], version into HTML output
```

If the build-time fetch fails, fall back to a hard-coded `FALLBACK_RELEASE`
constant. The build does not fail.

### Release data (runtime, optional refresh)

After client-side mount, refetch the same endpoint and update state. Errors are
swallowed silently; build-time data remains authoritative.

### i18n

Three JSON dictionaries, loaded server-side via the lang segment, no runtime
i18n library. Type checked via TypeScript const assertion of the English
dictionary; other dictionaries have the same shape enforced.

```ts
type Dict = typeof en;
const dictionaries: Record<Locale, Dict> = { en, vi, zh };
```

## SEO

Each `[lang]/page.tsx` emits:

- `<html lang="en|vi|zh-Hans">`
- Canonical link
- Three `<link rel="alternate" hreflang>` siblings + `x-default`
- OG meta (image: `/SmoothScroll/assets/og-image.png` — real PNG, replaces SVG)
- JSON-LD `SoftwareApplication`
- JSON-LD `FAQPage` (questions/answers from the same i18n dictionary)
- `<title>` and `<meta name="description">` from the dictionary

`sitemap.xml` is generated at build time by Next.js `app/sitemap.ts`.
`robots.txt` static, references the sitemap.

The Google Search Console verification file (`googleb5a10d9504de3274.html`) is
preserved as `public/googleb5a10d9504de3274.html`.

## Accessibility

- Skip-to-content link
- Semantic landmarks
- All interactive components keyboard-navigable; tab order matches visual order
- All images have explicit `width` and `height` (CLS prevention)
- All non-decorative images have `alt`; decorative ones have `alt=""`
- WCAG 2.2 AA color contrast verified for both themes
- `prefers-reduced-motion: reduce` disables Framer Motion transforms
- Lightbox dialog uses Radix Dialog primitive

## Testing strategy

| Layer | Tool | Coverage |
|---|---|---|
| Unit (utility logic) | Vitest | `useDownloadUrl`, `os.ts`, `github.ts` parsing |
| Component | Vitest + Testing Library | DemoScroll toggle, TrayPreview interactions, ExitIntentModal once-per-session |
| Visual regression | Playwright | 5 viewports × 2 themes × 3 langs (snapshot hero + tray preview only) |
| Accessibility | axe-core in Playwright | Zero serious/critical violations |
| Lighthouse | CI step | Perf ≥ 95, A11y ≥ 95, SEO 100 |
| Build smoke test | CI | `next build` exits 0, `out/` has 3 lang folders, sitemap.xml present |

Coverage target: 80% on `lib/**`. Components under 60% acceptable due to
visual-regression dominance.

## Performance budget

- LCP < 2.0s on slow 3G simulation
- CLS < 0.05
- INP < 150ms
- First-load JS < 120kb gzipped
- Hero hydration boundary kept tight (DemoScroll is the only client component
  in the hero)

## Migration plan

1. Build `landing/` in a feature branch (`landing-v3`)
2. Workflow `deploy-landing.yml` is forked into a `deploy-landing-v3.yml` that
   targets a preview branch (`gh-pages-preview`)
3. Verify v3 on preview URL, then merge `landing-v3` to master
4. Update `deploy-landing.yml` to point at `landing/` build output
5. Delete v2 source files from `docs/landing/` in the same merge commit
6. If post-merge regression: `git revert <merge-sha>` restores v2
